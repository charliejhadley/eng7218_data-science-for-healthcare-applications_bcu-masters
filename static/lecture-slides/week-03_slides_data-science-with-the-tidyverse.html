What is the tidyverse?

The tidyverse is two things

  • A collection of packages that are designed to work together really well.

  • A collection of packages for all stages of the data science workflow.

Source: R for Data Science1
  • An opinionated framework for how to work with data.

Some people2 would describe the tidyverse as being an alternative to using base R.

It‚Äôs not.

The tidyverse is a useful approach to working with data that has a large ecosystem of tools. We‚Äôll use it to move quickly.

After we‚Äôve been using the tidyverse for a while I‚Äôll come back to this subject and provide further context.

Understanding our object

The datasets embedded into {tidyverse} packages and those generated by reading in data files with {readr}, {readxl} and {haven} are objects known as ‚Äútibbles‚Äù.

It‚Äôs important to understand how this relates to and is different from data.frame.

data.frames (I)

The data.frame is R‚Äôs general purpose rectangular data store, it‚Äôs therefore the data structure required to build {ggplot2} charts.

The {datasets} package has a number of built-in data.frames, for instance:

data.frames (II)

Because they‚Äôre rectangular, data.frames have rows and columns which we can extract separately.

We can print the dimensions of a data.frame with dim()

dim(quakes)
[1] 1000    5

In Base R there are two ways to extract columns:

  • The $ operator allows us to extract columns via their name, with autocompletion:
quakes$mag
  • The more flexible [ operator allows us to extract both rows and columns, including by their index
quakes[, "mag"]
quakes[, 1]

Little Bunny Foo Foo

To introduce pipes, we‚Äôre going to borrow an example from Hadley Wickham:

Went hopping through the forest

Let‚Äôs create an instance of a bunny called foo_foo

Now let‚Äôs write the poem out as code:

  scoop_up(
    hop_through(foo_foo, forest),
    field_mouse
  ),
  head
)

Understanding our code

In order to understand what our code does, we need to parse it:

  • Find the deepest expression (the first thing that happens)

  • Work backwards (or up) the code

bop_on(
  scoop_up(
   hop_through(foo_foo, forest),
    field_mouse
  ),
  head
)

This is exactly counter to the order of operations in the original poem.

Piping little bunny foo foo

Let‚Äôs instantiate a bunny called foo_foo

foo_foo <- little_bunny()

Now write the same code as before but using pipes:

foo_foo %>%
  hop_through(forest) %>%
  scoop_up(field_mouse) %>%
  bop_on(head)

The order we read operations is exactly the same as the order in which the operations happen!

Comparing the two

Independent of pipes, we create ourselves a little bunny:

foo_foo <- little_bunny()

Now comparing the two code samples, the one with pipes is easier to parse by eye.

bop_on(
  scoop_up(
    hop_through(foo_foo, forest),
    field_mouse
  ),
  head
)
foo_foo %>%
  hop_through(forest) %>%
  scoop_up(field_mouse) %>%
  bop_on(head)

Admitedly, this doesn‚Äôt explain what %>% actually does!

Simpler %>% example

The pipe operator takes the left-hand side of your expression and inserts it into the first argument of the right-hand side of the expression:

"cats" %>% rep(4)

There‚Äôs nothing special about rep, it‚Äôs %>% which is doing the work.

%>% is an example of what‚Äôs called syntactic sugar it makes code easier to write/read.

Pushing the pipe further

In some cases you don‚Äôt want the left-hand side in the first argument, you can explicitly shove it somewhere else by using a period .
